# User Journeys

Known use cases WatchThat is built to serve. Each file is one journey — a real
task somebody does today, how WatchThat helps now, and where we want it to go.

Purpose:
- Keep product decisions anchored to real workflows, not abstract features.
- When something on the roadmap is unclear, a journey here should justify it.
- When a journey stops being served, update or remove its file — don't let it
  decay into aspirational fiction.

## Format

Each journey lives in its own file, named after the workflow (e.g.
`job-board-monitoring.md`). Keep it short. Suggested sections:

- **Persona** — who this is (one line)
- **Today, without WatchThat** — the manual workflow and its friction
- **With WatchThat (today)** — what we already support
- **Future** — what we'd need to add to serve this fully (link to roadmap
  versions in `README.md` when applicable)
